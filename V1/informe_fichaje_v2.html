Este informe evalúa el estado actual del sistema de fichaje por ubicación (V1) con el objetivo de planificar una V2 sólida y segura. El proyecto presenta una base técnica bien estructurada con separación limpia de responsabilidades, pero acumula varias vulnerabilidades de seguridad relevantes y carece de funcionalidades clave para el usuario final. A escala de menos de 50 empleados, el stack actual (SQLite, Express, Flutter) es completamente válido y no requiere cambios de infraestructura.

01

Fortalezas de V1

Lo que V1 hace bien y debe preservarse en V2.

🔐

Firma criptográfica Ed25519

Cada fichaje se firma en el dispositivo antes de enviarse. El backend verifica la firma. Esto garantiza no-repudio y previene la manipulación del fichaje en tránsito.

📐

Separación de responsabilidades

Backend, app móvil y panel admin son módulos independientes con contratos claros (API REST). El código de cada módulo es fácil de leer y modificar de forma aislada.

🌍

Doble verificación GPS

La ubicación se comprueba tanto en el cliente (mejor UX) como en el servidor (seguridad real). El algoritmo Haversine está correctamente implementado en ambos lados.

⏱️

Protección anti-replay

El backend valida que el timestamp del fichaje esté dentro de ±5 minutos de la hora real del servidor, impidiendo el reenvío de requests interceptados.

🐳

Docker listo para producción

Multi-stage build correcto, volumen persistente para la BD, variables de entorno por .env. El despliegue es reproducible desde el primer día.

📱

Cobertura multiplataforma

Flutter compila a Android, iOS, Windows y Linux desde el mismo código. El modo escritorio (verificación por IP) está correctamente diferenciado del móvil (GPS).

🔔

Notificaciones push en tiempo real

El sistema VAPID/Web Push está integrado y funcional. Las suscripciones caducadas se limpian automáticamente. El admin recibe alertas instantáneas de fichajes.

📊

Panel admin completo

Filtros combinados, paginación configurable, exportación CSV con BOM Unicode, indicador visual de firma válida/inválida. Cubre el flujo administrativo básico.

02

Análisis de seguridad

10 hallazgos ordenados por severidad. Los marcados como Crítico / Alto deben resolverse antes de ir a producción real.

Crítico admin_panel

JWT almacenado en localStorage — vulnerable a XSS

localStorage.getItem('token') es accesible por cualquier script JS en la misma página. Si un atacante inyecta código en el panel, obtiene el token de admin sin fricción.

→ Migrar a cookie httpOnly + SameSite=Strict. El backend envía la cookie en el login y el panel no necesita gestionar el token manualmente.
Alto server.js

CORS completamente abierto — acepta cualquier origen

app.use(cors()) sin configuración permite peticiones desde cualquier dominio. Un atacante puede construir una página que haga peticiones autenticadas al backend.

→ app.use(cors({ origin: ['https://mipanel.empresa.com'], credentials: true }))
Alto routes/auth.js

Sin rate limiting — ataque de fuerza bruta posible

El endpoint POST /auth/login no tiene ningún límite de intentos. Un atacante puede probar miles de contraseñas por segundo contra cualquier cuenta.

→ npm install express-rate-limit — aplicar límite de 10 intentos / 15 min por IP en /auth/login.
Alto utils/ip.js

X-Forwarded-For aceptado sin validación — IP spoofing

La función getClientIp() confía ciegamente en el header x-forwarded-for. Si el backend no está detrás de un proxy de confianza, cualquier cliente puede falsificar su IP y fichar como si estuviera en la oficina.

→ Configurar express: app.set('trust proxy', 1) — solo confiar en el primer proxy. Sin proxy, eliminar la lectura de x-forwarded-for.
Medio seed.js

Credenciales de admin por defecto hardcodeadas

admin@empresa.com / admin1234 son las credenciales del primer admin. Es fácil olvidar cambiarlas antes de un despliegue real, especialmente en entornos CI/CD.

→ Leer email y password desde variables de entorno en seed.js. Rechazar arranque si las variables no están definidas.
Medio routes/push.js

Búsqueda de push subscription con LIKE sobre datos JSON

WHERE datos LIKE '%${endpoint}%' es frágil: si el endpoint contiene caracteres especiales de SQLite LIKE (%, _), la búsqueda falla o da falsos positivos.

→ Añadir columna endpoint TEXT UNIQUE a push_suscripciones. Buscar por esa columna exacta en lugar de LIKE sobre el JSON.
Medio config.dart

URL del backend hardcodeada en el código fuente

http://192.168.1.100:3000 está en el código compilado. Cambiar el servidor implica recompilar y redistribuir la app. Además, usa HTTP plano en lugar de HTTPS.

→ Usar --dart-define=API_URL=... en build. En producción, forzar HTTPS. Validar que la URL no sea la IP de desarrollo.
Bajo routes/auth.js

Sin mecanismo de refresh token

El JWT dura 8 horas y no puede renovarse. Si un empleado trabaja un turno largo o la app queda abierta, al expirar hay un logout abrupto sin aviso previo.

→ Emitir un refresh token de larga duración (httpOnly cookie). Añadir POST /auth/refresh para renovar el access token silenciosamente.
Bajo routes/empleados.js

Eliminar empleado no elimina sus fichajes (datos huérfanos)

DELETE FROM empleados WHERE id = ? no borra los fichajes asociados. La clave foránea no tiene ON DELETE CASCADE, por lo que los fichajes quedan sin empleado referenciado.

→ Añadir ON DELETE CASCADE en la FK de fichajes, o implementar soft-delete (campo activo=0) para preservar historial con fines legales.
Info server.js

Sin middleware global de manejo de errores

Si un handler de Express lanza una excepción no capturada, Express devuelve un HTML de error por defecto que puede exponer rutas internas o el stack trace.

→ Añadir al final: app.use((err, req, res, next) => res.status(500).json({ error: 'Error interno' }))
03

Arquitectura y calidad de código

El código es limpio y legible pero acumula deuda en testabilidad, observabilidad y mantenibilidad.

🧪

Sin tests (cero cobertura)

No hay ningún fichero de test en el proyecto. Las funciones críticas como verificarFirma(), estaEnEmpresa() o los endpoints de fichaje no tienen cobertura automatizada. Cualquier refactor de V2 es un cambio a ciegas.

📋

Sin logging estructurado

Solo hay console.log y console.error. En producción esto dificulta el diagnóstico de errores, imposibilita alertas automatizadas y no hay rastro de quién hizo qué. Se recomienda pino o winston.

🔄

Migraciones sin versionado

El sistema de migraciones es un array de ALTER TABLE dentro de un try/catch. Funciona pero no registra qué migraciones se han ejecutado. Al añadir más columnas en V2 esto se volverá frágil. Considerar better-sqlite3-migrate.

⚙️

config.dart duplica el backend

empresaLat, empresaLon y radioMetros deben estar sincronizados manualmente entre .env del backend y config.dart de Flutter. Una deriva entre ambos provoca inconsistencias silenciosas.

📡

Sin versionado de API

Las rutas son /auth/login en lugar de /v1/auth/login. En V2, si se rompe la compatibilidad con la app V1 instalada en dispositivos, no hay forma de distinguir qué versión llama a qué endpoint.

📄

Sin documentación de API

El README incluye una tabla básica pero no hay especificación OpenAPI/Swagger. Dificulta la incorporación de nuevos desarrolladores y la generación automática de clientes HTTP.

04

UX y funcionalidades — Brechas actuales

Lo que los usuarios esperan y V1 no cubre.

📅

El empleado no ve su propio historial

La app solo permite fichar. El empleado no puede consultar sus entradas y salidas, horas trabajadas ni descargar su propio registro. Genera dependencia del administrador para cualquier consulta.

🟢

Sin dashboard de presencia en tiempo real

El admin no puede ver de un vistazo quién está dentro de la empresa en este momento. Tiene que filtrar la tabla de fichajes manualmente para inferir el estado actual.

✏️

Sin corrección manual de fichajes

Si un empleado olvida fichar o hay un error de GPS, el admin no puede insertar ni corregir un fichaje. No existe ningún flujo de justificación o edición manual.

🏠

Solo entrada/salida — sin tipos de ausencia

No hay forma de registrar vacaciones, bajas, teletrabajo o días de asunto propio. El sistema no puede distinguir entre "no fichó porque faltó" y "no fichó porque trabajó desde casa".

🔑

Intercambio de clave pública manual y frágil

El empleado copia 44 caracteres base64 de la app y se los pasa al admin, que los pega en un campo de texto. Es propenso a errores, no hay validación visual y no existe flujo de cambio de dispositivo automatizado.

📊

Sin estadísticas ni informes de horas

No hay cálculo de horas trabajadas por día/semana/mes, resumen de puntualidad ni exportación en formato compatible con software de RRHH. Solo existe la exportación CSV básica.

🔒

Sin recuperación de contraseña

Si un empleado olvida su contraseña, solo el admin puede resetearla manualmente. No hay flujo de email de recuperación.

📱

Panel admin no optimizado para móvil

El panel HTML tiene un layout de tabla que no es usable en pantallas pequeñas. Un admin gestionando desde el móvil tiene una experiencia degradada.

05

Roadmap V2 — Funcionalidades propuestas

Ordenadas por impacto real para el usuario. Las etiquetas indican qué capa del sistema requiere trabajo.

Fase 1
Seguridad y base
🍪
Migración de JWT a cookie httpOnly
Eliminar la vulnerabilidad XSS del panel admin. El token viaja en cookie segura y no es accesible desde JavaScript. Incluye implementación de refresh token silencioso.
backend admin panel seguridad crítica
🛡️
Rate limiting + CORS restringido
Protección de fuerza bruta en login (10 intentos/15 min por IP) y lista blanca de orígenes CORS. Configuración desde variables de entorno.
backend seguridad alta
📋
Logging estructurado + gestión de errores global
Integrar pino para logs en JSON. Añadir middleware global de errores en Express. Los logs incluyen empleado_id, endpoint, IP y duración de cada request.
backend
Fase 2
Autoservicio empleado
📅
Historial personal en la app Flutter
Nueva pantalla en la app donde el empleado ve sus fichajes del mes con horas trabajadas calculadas, filtro por semana y exportación PDF personal. Nuevo endpoint GET /fichajes/mios.
flutter backend
🔐
Intercambio de clave pública automatizado
Tras el primer login, la app envía automáticamente su clave pública al backend si el empleado aún no tiene una. El admin recibe notificación push y la aprueba con un solo clic.
flutter backend admin panel
🔑
Recuperación de contraseña por email
Flujo estándar: solicitar reset → email con enlace firmado (JWT de un solo uso, 30 min) → formulario de nueva contraseña. Requiere integración con proveedor SMTP (nodemailer + SendGrid/Resend).
backend flutter admin panel
Fase 3
Gestión avanzada
🟢
Dashboard de presencia en tiempo real
Sección en el panel admin con tarjetas de empleados mostrando si están "dentro", "fuera" o "sin actividad hoy". Se actualiza vía push notification al recibir fichajes. Incluye contador de asistencia diaria.
admin panel backend
✏️
Corrección y inserción manual de fichajes
El admin puede añadir fichajes pasados (empleado olvidó fichar) o corregir hora/tipo. Cada corrección queda registrada con quién la hizo y cuándo (campo corregido_por en la BD).
admin panel backend
🏠
Tipos de ausencia y teletrabajo
Nuevos tipos de registro: vacaciones, baja médica, teletrabajo, asunto propio. El empleado solicita desde la app; el admin aprueba. Los informes de horas los excluyen o incluyen según configuración.
flutter backend admin panel
📊
Informes de horas trabajadas
Cálculo automático de horas por empleado/semana/mes. Identificación de días con entrada pero sin salida. Exportación en Excel (.xlsx) con formato compatible con nóminas. Gráficas de asistencia en el panel.
admin panel backend
🏢
Soporte de múltiples sedes
Tabla de sedes con nombre, coordenadas GPS y radio configurables. Cada empleado se asigna a una sede. El fichaje valida contra la sede del empleado, no una coordenada global.
backend admin panel flutter
